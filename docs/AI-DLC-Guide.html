1. The Problem We Are Solving

There are actually two distinct problems, and this workspace solves both. It helps to keep them separate in your head.

Problem A — AI loses context across repositories

When a system is split across many repos, AI coding tools lose the big picture. By default, if you open a single service repo, the AI only knows about that service. It doesn't know what other services exist, how they talk to each other, what contracts they share, or what decisions have already been made. The results:

  • The AI suggests changes that quietly break other services.
  • Logic gets duplicated because the AI doesn't know it already exists elsewhere.
  • API shapes drift inconsistent because the AI can't see the shared contracts.
  • Refactors miss inter-service dependencies entirely.

The naive fix — keeping a separate "context repo" next to the service repos — doesn't work. You can't guarantee everyone cloned it, put it at the right path, or kept it in sync. The AI still won't automatically pick it up when you open a service folder.

Problem B — AI development without discipline ("vibe coding")

Even with full context, letting an AI generate code from a one-line prompt produces work with no plan, no record of decisions, and no checkpoints. Requirements are assumed rather than confirmed, design choices are invisible, and nobody can later answer "why was it built this way?" This is fast on day one and expensive every day after.

The two-part answer. A workspace repo with git submodules solves Problem A (context). The AI-DLC methodology solves Problem B (discipline). The rest of this document explains each, and how they fit together in this repo.

2. Solution, Part 1 — The Workspace + Git Submodules

We use a single workspace repo that everyone clones. It does not contain service code directly — it stitches the independent service repos together as git submodules. Open the workspace root in your IDE (or run Claude Code from it) and the AI can see every service at once. One clone, full context.

Each service repo stays independent — its own git history, its own CI/CD, its own deployments. The workspace just records a pointer (a commit hash) to a known-good version of each service.

    └── api-gateway/          ← submodule → ArfaMujahid/api-gateway

How the pointer model works (the key idea)

The workspace doesn't store the service code. Each submodule entry is just a commit hash pointing at a specific state of the child repo. That gives you:

  • Services that are independently deployable and versionable.
  • Different teams owning different repos.
  • A workspace that always reflects a known-good combination of service versions.

Updating a service is therefore two moves: push the service repo, then bump the pointer in the workspace. (Section 3 covers this in detail — it's the part people get wrong.)

One submodule is not ours

aidlc-workflows/ points at the upstream awslabs/aidlc-workflows repo, not one of our repos. We pin it at a known commit and only move that pointer deliberately. We never push to it — we just track the version of the methodology we're using.

Cloning the right way

# --recurse-submodules pulls the service repos too
git clone --recurse-submodules git@github.com:ArfaMujahid/AISLCTesting.git

# Forgot the flag and services/ came up empty? Fix it with:
git submodule update --init --recursive

3. Working With Submodules (the how-to)

Every submodule change touches two repos: the service repo (where the code lives) and the workspace repo (which stores the pointer). Nothing automatically links them — you make two deliberate commits, and order matters.

The two-repo mental model

Where you commitWhat you're committingWhere it goes
services/user-serviceactual code changesArfaMujahid/user-service
AISLCTesting (workspace root)the updated submodule pointer (a commit hash)ArfaMujahid/AISLCTesting

The golden rule

Always push the service repo first, then bump the workspace pointer. If you reverse it, other developers pull a commit hash that doesn't exist on the remote yet — and their clone breaks.

Step 1 — commit & push inside the submodule

cd services/user-service
git checkout main          # avoid detached HEAD — see the footgun below
# ... make your edits ...
git add .
git commit -m "your change"
git push origin main       # → pushes to ArfaMujahid/user-service

At this point your code is safe on the service remote, but the workspace still points at the old commit — it has no idea anything changed.

Step 2 — bump the pointer in the workspace

cd ../..                   # back to the workspace root
git add services/user-service   # stages the NEW commit hash, not the code
git commit -m "bump user-service to <short-hash>"
git push                   # → pushes to ArfaMujahid/AISLCTesting

That second commit contains no service code — just the moved pointer. Confirm with git diff --cached and you'll see -Subproject commit abc123… / +Subproject commit def456….

The detached-HEAD footgun

Submodules check out a specific commit, not a branch, so by default you're on a "detached HEAD." If you commit there without first running git checkout main, your commit isn't on any branch and git push won't know where to send it — the work appears to vanish. Always git checkout main inside the submodule before editing.

Common submodule footguns

ProblemWhat happensFix
Cloned without --recurse-submodulesservices/ folders are emptygit submodule update --init --recursive

4. Solution, Part 2 — What AI-DLC Is and Why

AI-DLC (AI-Driven Development Life Cycle) is an intelligent, adaptive software-development workflow from AWS Labs. It turns "ask the AI to build something" into a structured lifecycle that maintains quality standards and keeps humans in control at every step.

The central bargain:

The AI does the work. Humans hold the gates. The AI analyses, plans, designs, and generates code — but it stops at every stage boundary and waits for explicit human approval before moving on. Every decision and every prompt is recorded.

What it gives you that raw prompting doesn't

  • Adaptive depth — a one-line fix gets a light touch; a production service gets full requirements, design, and NFR treatment. The workflow adapts to the work, not the other way around.
  • Durable artifacts — requirements, user stories, designs, and plans are written to files (aidlc-docs/), so decisions survive context resets and team handoffs.
  • Approval gates — the AI never silently barrels ahead; you review and approve each stage.
  • A complete audit trail — every prompt and response is logged with timestamps in audit.md.
  • A single source of truth — design documents drive code generation, so the docs and the code stay in sync.

The adaptive principle

Not every stage runs every time. Some stages are ALWAYS executed; the rest are CONDITIONAL — the AI decides whether they add value based on your intent, the existing codebase, and the complexity and risk of the change. There are no fixed sequences; stages run in the order that makes sense for the task.

6. How You Interact With It — the Gate Model

This is the part that feels different from normal AI chat. AI-DLC does not ask clarifying questions inline and then guess. It writes questions to a file, waits, and stops at every stage boundary.

The Question → Doc → Approval flow

1. AI writes a question file
and stops		 → 2. You fill in [Answer]: tags
a durable, reviewable record		 → 3. You say "answers ready, re-read & proceed" → 4. AI validates & generates the next artifact

Questions are multiple-choice. You answer in the file itself:

## Question: Deployment model
Where will this service be deployed?

A) AWS Lambda (serverless)
B) AWS ECS Fargate (containerized)
C) Existing on-premises infrastructure
X) Other (describe after the [Answer]: tag)

[Answer]: B — containerized; we already run ECS in this account

Tips that work well: add a label next to the letter, include a one-line justification, combine options when you mean both (B and C), and use X freely when nothing fits rather than forcing a wrong answer.

Approval gates

At the end of each stage, AI-DLC presents a standardized two-option completion message:

Request ChangesApprove & Continue
Ask for modifications before moving on. The AI revises and re-presents.Accept the output and advance to the next stage.

Read the generated artifact before you approve. Design documents are the source of truth for code generation — mistakes that slip through a gate are much harder to fix later.

Context management — clear at the gates

Each gate is a natural place to start a fresh context. The AI's work is already saved to files, so a clean reset is low-risk: the next context loads the relevant artifacts from disk and continues without dragging along accumulated noise. If your tool offers a "compact context" prompt mid-workflow, decline it — compaction loses more than a clean reset. To resume:

Go to aidlc-docs/aidlc-state.md, find the first unchecked item,
then go to the corresponding plan file and resume from that point.

The two files that track everything

FileRole
aidlc-docs/aidlc-state.mdProgress & gate checklist — which stages ran, which were skipped, what's next.
aidlc-docs/audit.mdComplete audit trail — every prompt and response, raw and timestamped (ISO 8601). Always appended, never overwritten.

7. The Core Discipline — Never Vibe Code

"Vibe coding" means directly editing generated code to make a quick fix — bypassing the design documents. It feels fast and creates drift: the design docs (the source of truth AI-DLC reads for every subsequent operation) no longer match what the code actually does. The next code-generation run, session resume, or teammate handoff then hits confusion and rework.

The rule: update the design first, then regenerate the code. Never fix the code directly.

The right way to make any change

StepWhat you do
1Describe the issue, touch nothing. "Do not update any documents yet. I've found issue X. Help me understand where it needs to be addressed."
2Fix the design document. "Update [design doc] to reflect the fix, then check whether requirements or user stories also need updating."
3Regenerate the affected code. "The design for [unit] changed — re-run code generation for the affected files only."

Two standing rules that prevent drift

Set these once at the start of Construction and they apply for the whole phase:

  • Back-propagation: "Every time you update a document, check whether the change impacts the requirements document and user stories, and prompt me if it does."
  • Design-first: "When you make a design decision during code generation, always make sure the documentation reflects it before proceeding."

One more practical note: ask exploratory questions safely by prefixing them with "Do not update any documents." — otherwise the AI may interpret a question as a change request and start editing.

8. How AI-DLC Is Wired Into This Repo

AI-DLC isn't copied into the workspace — it's pulled in as a submodule, the same mechanism as the services, so we track an exact version of the methodology.

PieceWhat it is
aidlc-workflows/Submodule → awslabs/aidlc-workflows. The steering rules the AI follows. External and pinned — we never push to it; we only move the pointer deliberately to adopt a new version.
aidlc-docs/The working artifacts AI-DLC produces (requirements, designs, state, audit). Tracked in git, so the decisions are versioned alongside the workspace.
CLAUDE.mdGlobal context the AI reads first — including the rule to follow AI-DLC and the cross-service rules.

Where code goes vs where docs go

Application code → the workspace root (inside the relevant service). Documentation → aidlc-docs/ only. Never put code in aidlc-docs/, and keep human-facing reports out of it too — files there are reloaded as design input on later stages and would inflate context.

Kicking off a session

Prepare two short inputs first — they dramatically cut the number of clarifying questions:

  • Vision Document — what you're building and why; MVP features in scope; what's explicitly out of scope; known open questions.
  • Technical Environment Document — language/version, framework, cloud + deploy model, test framework, a prohibited-libraries table (with reason + alternative), and example code patterns for a typical endpoint, function, and test.
I want to start a new project. Please read [vision document] and
[technical environment document], then begin the AI-DLC workflow.

The non-negotiables

  • Run the AI from the workspace root, never from inside a service folder — that's what gives it cross-service context.
  • Push the service repo before bumping the workspace pointer.
  • Update shared-contracts/ before changing any API shape or event.
  • Never vibe code — update the design, then regenerate.
  • Read each artifact before approving a gate.
